# Learnings

- Frontend foundation uses `vitest` + `@testing-library/react` for a single app-shell render test and `@playwright/test` for one deterministic smoke journey against the root route.
- Keeping the initial seller shell as static server-rendered content lets `npm run build`, `npm test`, and `npm run test:e2e` pass without introducing auth, API, or client-side state assumptions.

- Importing `describe`, `it`, and `expect` from `vitest` keeps editor/LSP test typing clean without widening global TypeScript assumptions; `npm run test:e2e` now clears legacy `test-results/` before and after Playwright runs.
- The backend frontend contract is a single JSON envelope of `{ success, message, data, trace_id }`; normalizing that once in the API layer keeps later feature work focused on typed `data` plus trace-aware error handling.
- Cookie-auth requests should build headers with `new Headers(...)` before passing them to `fetch`; spreading a `Headers` instance into a plain object drops forwarded cookies and breaks server-side `/auth/me` bootstrap flows.

- The seller listings endpoint for this route is `/auth/me/listings`, and it returns paginated data under the shared envelope; the dashboard can stay server-rendered by reading `data`, `total`, `page`, `limit`, and `total_pages` directly from the normalized response.
- Keeping `/dashboard` resilient requires handling two separate failure layers: `/auth/me` bootstrap can fail before authorization is known, while `/auth/me/listings` can independently return `401`, empty data, or transport errors; rendering first-class states for each keeps browser smoke tests stable even without a live backend.
- Seller listing create/edit flows can stay lightweight without a form library by keeping one controlled client component that fetches public categories plus optional listing hydration, then serializes directly into the backend `CreateListingRequest` or `UpdateListingRequest` shape.
- The category API returns nested roots with `children`; flattening them into select options like `Parent / Child` preserves backend-driven taxonomy while staying easy to assert in Vitest and Playwright without a live backend.
- Seller image mutations work cleanly as small button-driven actions (`upload`, `set primary`, `move earlier/later`, `delete`) so long as the UI replaces only the hydrated listing record from each backend response instead of inventing an optimistic image store.
- Avoid nesting an image-upload `<form>` inside the listing edit `<form>`; keeping upload as a button-driven action prevents hydration warnings and accidental submission crossover while still letting the backend response stay authoritative.
- Listing-form API tests need explicit coverage for category flattening, specification normalization, and trace-aware error formatting so regressions in backend-envelope adaptation are caught before component tests.
- Playwright coverage is more reliable when it asserts request payloads/methods during create, update, and image mutations while stubbing `/auth/me` + `/auth/me/listings` bootstrap; this validates both seller UX and contract-level correctness without a live backend.
- `frontend/AGENTS.md` and root `AGENTS.md` were stale after T1-T5. They now need to describe active seller routes (`/dashboard`, `/dashboard/listings/new`, `/dashboard/listings/[listingId]/edit`) instead of scaffold-only wording.
- Frontend guidance should explicitly call out the active verification stack, `vitest` + Testing Library for unit/component tests and Playwright for browser smoke tests, because these are now part of routine frontend validation.
- Cookie-auth expectations are an implementation detail now, not future guidance. Docs should mention `credentials: "include"` in API requests and server-side cookie forwarding through `cookies()` helpers for `/auth/me` and `/auth/me/listings`.
- Dashboard subtree auth is safest at `app/dashboard/layout.tsx`; redirecting unauthenticated requests before the shell renders removes mixed guest/seller states from every nested seller route at once.
- Seller edit hydration should not call the public `/api/listings/:id` read path because it increments `view_count`; server-side preload from seller-scoped inventory avoids that side effect and keeps edit state owner-aware.
- Final verification confirmed the remaining unchecked plan items were acceptance sub-items only: `npm run lint`, `npm run build`, `npm test`, and `npm run test:e2e` all pass in the seller worktree, and the auth/bootstrap/dashboard/listing/image/docs criteria are satisfied by the current implementation.
- Running the buyer-facing browse/detail flow locally without the Go service causes the public page to render zero results and the client save button to stay on `/listings/test` (no `/login` redirect) because every protected API call (`/api/categories`, `/auth/me`) points at `http://127.0.0.1:8080` and fails with CORS when the backend is absent.
